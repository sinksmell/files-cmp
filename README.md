## 📖 项目简介

**Files-CMP** 是一个高性能的分布式文件对比工具，专为解决大规模文件对比场景而设计。当两台机器各自存储千万级小文件，且仅有少量文件存在差异时，本工具能够在低带宽、高延迟的网络环境下，以最小的内存占用和最快速度找出差异文件。

### 适用场景

- 🖥️ 两台机器各有大量文件（如 1000 万+）
- 📊 仅有极少数文件存在差异（如 3-5 个）
- 🌐 网络条件较差（如几百 MB 需要传输 10 小时）
- 💾 磁盘性能良好（SSD）
- 🔄 支持增量同步

---

## 🎯 核心特性

- ✅ **IBLT 算法**: 采用 Invertible Bloom Lookup Table，专为集合调和设计，在极稀疏差异场景下表现卓越
- ✅ **低带宽优化**: 仅需传输少量哈希数据即可定位差异，大幅减少网络传输量
- ✅ **低内存占用**: 流式处理，适合大规模文件对比
- ✅ **智能对比**: 
  - 文本文件：逐行分析，高亮显示差异
  - 二进制文件：对比前 512 字节
- ✅ **增量同步**: 支持检测并同步增量变动
- ✅ **彩色输出**: 终端彩色高亮显示对比结果

## 🏗️ 架构设计

### 对比策略

```
┌─────────────┐                    ┌─────────────┐
│   Client    │ ◄──── HTTP ────►   │   Server    │
│  (客户端)    │                    │  (服务端)    │
└─────────────┘                    └─────────────┘
      │                                   │
      ▼                                   ▼
 1. 获取文件列表                      1. 初始化文件列表
 2. 计算文件 MD5                      2. 构建 IBLT
 3. 发送 IBLT                         3. 构建 IBLT
 4. 接收差异文件列表                  4. XOR 运算并解码
 5. 发送差异文件                      5. 详细对比
```

### 工作流程

#### 第一阶段：IBLT 集合调和

1. **文件列表获取**: 使用 `ls` 命令获取文件名列表（自动按字母序排列）
2. **计算文件哈希**: 遍历文件集合，计算每个文件的 MD5 值
3. **构建 IBLT**: 将 `MD5(文件名):MD5(文件内容)` 作为键值对插入 IBLT
4. **网络传输**: Client 和 Server 互相发送各自的 IBLT（或仅发送差值 IBLT）
5. **XOR 运算**: 双方对 IBLT 进行异或操作，得到包含差异的 IBLT
6. **解码**: 使用剥洋葱算法（Peeling Decoder）从差异 IBLT 中恢复出所有差异文件

#### 第二阶段：文件内容对比

- 📝 **文本文件**: 分析独特内容，高亮显示变化
- 🖼️ **二进制文件** (JPG, MP4 等): 对比前 512 字节

## 🚀 快速开始

### 环境要求

- Go 1.x 或更高版本
- Linux / macOS / Windows

### 安装依赖

```bash
go get github.com/astaxie/beego
```

### 启动服务端

```bash
go run main.go
```

服务端将：
- 初始化 `./file/` 目录下的文件列表
- 自动进行文件分组，生成 `./group/` 目录下的分组文件
- 启动 HTTP 服务（默认端口 8080）

### 启动客户端

```bash
cd client
go run client.go
```

客户端将：
- 自动对比 `./file/` 文件夹下的所有文件
- 与服务端进行 IBLT 集合调和
- 找出并上传差异文件进行详细对比

## 🔧 配置说明

### 核心常量 (`models/comm.go`)

| 常量 | 默认值 | 说明 |
|------|--------|------|
| `FILE_PATH` | `./file/` | 小文件存放路径 |
| `CMP_SIZE` | `512` | 非文本文件比较字节数 |
| `IBLT_CELLS_FACTOR` | `3` | IBLT cell 数量 = 预期差异数 × factor |

### 响应码

| 代码 | 含义 |
|------|------|
| `0000` | 文件 MD5 相同 |
| `0001` | 文件处理成功 |
| `1000` | 文件 MD5 不同 |
| `2000` | 请求错误 |
| `3001` | 文件保存错误 |
| `3002` | 文件对比错误 |

## 💡 性能优势

| 传统方法 | Files-CMP (IBLT) |
|---------|------------------|
| 全量传输所有文件 | 仅传输 IBLT（几 KB）+ 差异文件 |
| 内存占用高 | 固定大小 IBLT，低内存 |
| 网络开销大 | O(1) 传输复杂度（与文件总数无关） |
| 速度慢 | 单次 XOR 运算 + 解码，快速定位 |

**示例**: 1000 万文件，仅 5 个差异
- 传统方法：需传输全部文件（可能数百 GB）
- Files-CMP：仅传输 IBLT（~2KB）+ 5 个差异文件（几 MB 到几十 MB）

### 为什么选择 IBLT？

Invertible Bloom Lookup Table (IBLT) 是一种概率数据结构，专为集合调和（Set Reconciliation）设计：

- **空间效率**: IBLT 大小仅与预期差异数量相关，与集合总大小无关
- **解码保证**: 当 IBLT 大小足够时，可以以极高概率成功解码所有差异
- **单次通信**: 双方只需交换一次 IBLT 即可完成调和
- **线性时间**: 插入、删除、解码均为 O(n) 时间复杂度

**数学原理**: 
- IBLT 由多个 Cell 组成，每个 Cell 包含计数器、键异或和、值异或和
- 通过"剥洋葱"算法（Peeling Decoder），从度数为 1 的 Cell 开始迭代恢复差异
- 理论证明：当 cell 数量 ≥ 3×差异数量时，解码成功率 > 99.9%

详细论文参考：[Invertible Bloom Lookup Tables](https://www.ics.uci.edu/~goodrich/teach/cs206/notes/IBLT.pdf)
